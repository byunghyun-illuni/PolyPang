# PRD — PolyPang (웹 실시간 멀티 게임) MVP

---

## 0. 문서 목적 & 범위

### 0.1 목적

* 이 문서는 **실시간 멀티 웹게임 “PolyPang”의 MVP**를 정의한다.
* 목표는:

  * 개발자가 이 문서만으로 **클라이언트/서버 구현을 시작**할 수 있고
  * 화면기획자가 이 문서를 참고해 **화면 플로우·컴포넌트**를 설계할 수 있도록 하는 것이다.

### 0.2 범위 (이번 버전에서 실제로 구현하는 것)

* 2~8명 멀티 플레이, **방 코드 기반 입장**
* **정N각형 경기장** 기반 생존 게임:

  * 각 플레이어: 정N각형의 한 변(Side)
  * 공(Ball) 1개 고정
  * 공을 패들로 막지 못한 플레이어는 **OUT**
  * 플레이어 OUT 시, 경기장이 **정(N-1)각형**으로 축소·재배치
* **공 속도 증가 규칙**: 패들이 공에 닿을 때마다 전체 속도 5% 증가
* **PolyPang 아이덴티티 연출**:

  * 공-패들 히트 시: 소형 Pang 이펙트
  * OUT 시: 대형 Pang 이펙트 + 경기장 변 파괴 연출
* 관전 모드: OUT 이후 게임 관전 + 기본 이모지 리액션
* 서버 authoritative 구조에서의 **판정/동기화** 기본 규칙

### 0.3 이번 버전에서 다루지 않는 것 (개념만, 구현 X)

> *구체 내용 언급 없이, “확장용 슬롯” 정도로만 남김.*

* 결승/특수 규칙 추가(특수 필드, 특수 모드 등)
* 아이템, 버프/디버프, 스킬
* 랭킹/시즌/계정 진척도/코스튬/상점
* 상세 소셜 기능(친구 목록, 초대 링크, 신고 등)

## 1. 게임 컨셉

### 1.1 한 줄 설명

> **정N각형 경기장에서 각 변을 맡은 플레이어들이 공을 튕겨내며 버티고, 공을 놓친 플레이어가 “Pang!”과 함께 탈락하면서 경기장이 점점 줄어드는 실시간 생존 핀볼 게임.**

### 1.2 핵심 특징

1. **경기장 축소 = 탈락 연출**

   * 플레이어가 OUT될 때마다 그 변이 “Pang!”하고 깨지며 사라지고,
     경기장이 즉시 정(N-1)각형으로 재형성된다.
2. **시간이 갈수록 빨라지는 공**

   * 공이 패들에 닿을 때마다 전체 속도가 5%씩 증가.
   * 라운드가 길어질수록 긴장감이 기하급수적으로 상승.
3. **룰은 단순, 감각은 아케이드**

   * 조작: 좌/우 이동만.
   * 하지만 각도, 속도, 튕김 패턴 때문에 실력과 운이 자연스럽게 섞임.

### 1.3 타깃 플레이 & 톤

* 친구들/지인과 함께 하는 **가벼운 파티게임**
* 1판 1~3분 정도의 짧은 플레이 타임
* 분위기: 밝고 경쾌, 과하게 진지하지 않은 **캐주얼 아케이드**

---

## 2. 기본 규칙 & 게임 루프

### 2.1 플레이어 수

* 최소 2명, 최대 8명
* 게임 시작 시 인원을 확정하고, 이후 중途 합류 없음 (MVP 기준)

### 2.2 한 판의 흐름 (상위 루프)

1. **방 생성 / 입장**
2. **로비**에서 인원 모임 & Ready
3. 방장이 Start → **카운트다운(3초)**
4. **인게임**:

   * 공 1개 발사
   * 패들로 공을 튕겨내며 생존
   * 패들이 공을 놓친 플레이어는 OUT
   * OUT 시 경기장 축소 (정N→정(N-1))
5. Alive가 1명 남으면 **게임 종료**
6. **결과 화면** → 로비로 돌아가기 or 방 나가기

## 3. Arena & Geometry (경기장 기하 스펙)

### 3.1 기본 개념

* Alive 플레이어 수 = N이라고 할 때, 경기장은 항상 **정N각형**.
* 각 플레이어는 자신에게 할당된 1개의 변(Side)을 가진다.
* N이 줄어들수록 경기장은 단계별로 정N각형 → 정(N-1)각형 → … → 정2각형(사실상 양 끝) → 마지막 1명 생존.

### 3.2 정N각형 정의 (수학적 스펙)

* 중심: C = (0, 0)
* 반지름: R (뷰포트/화면 크기에 따라 비율로 산정)

  * 예시: `R = (min(화면폭, 화면높이) * 0.35 ~ 0.4)`
* i번째 Side의 **중심 각도** θ_i (i = 0 ~ N-1):

  * θ_i = 2π * i / N (라디안 기준)
* 각 Side의 길이 L:

  * L = 2 * R * sin(π / N)

### 3.3 Side & Player 매핑

* 플레이어 리스트 순서를 기준으로,
  i번째 Alive 플레이어를 Side i에 배치 (시계 혹은 반시계 방향으로 고정).
* OUT이 발생하면:

  * 해당 플레이어는 리스트에서 제거.
  * 남은 Alive 플레이어를 0~M-1로 다시 인덱싱하고
    정M각형 상에 **균등 간격으로 재배치**.

### 3.4 리메시(재배치) 애니메이션

* OUT 직후 ~ 약 0.5초 슬로우 연출 구간:

  1. OUT Side가 깨져서 사라짐.
  2. 나머지 Side들이 중심을 향해 모이면서,
     각자의 새 각도 위치(정M각형)로 부드럽게 이동.
* 이때 공은 잠깐 속도가 느려진 상태로 유지(슬로우),
  리메시 끝과 동시에 정상 속도로 복귀.

## 4. Paddle (패들) 규칙

### 4.1 위치 & 길이

* 각 플레이어는 자기 Side 위에 1개의 Paddle을 가진다.
* Paddle 중심은 Side의 중앙을 기준으로 위치.
* Paddle 길이:

  * `paddle_length = α * L`
  * α 초기값 제안: 0.3 (Side 길이의 30%)

### 4.2 이동 범위

* Paddle은 해당 Side를 따라 직선상으로만 이동 가능.
* 이동 가능한 범위:

  * Side 길이의 β 구간만 사용 (양 끝에 여유 공간 존재)
  * `move_range = β * L`, β 초기값: 0.6
* 즉, Side의 정중앙을 기준으로 좌우로 `move_range / 2` 만큼만 이동 가능.

### 4.3 조작 입력 & 물리

* 입력:

  * 모바일:

    * 화면 좌측 터치: 왼쪽 이동
    * 화면 우측 터치: 오른쪽 이동
  * PC:

    * A / ← : 왼쪽
    * D / → : 오른쪽
* 이동 방식:

  * 키/터치 입력 유지 시 **가속도** 형태로 속도 증가
  * 최대 속도 제한 존재 (튜닝값)
  * 입력 없음 시 서서히 감속 → 정지
* 서버/클라:

  * 클라이언트에서 입력을 서버에 전송
  * 최종 Paddle 위치는 서버 기준이지만, 클라는 **예측 렌더링** 가능

## 5. Ball (공) 규칙

### 5.1 기본 스펙

* Ball 개수: **항상 1개** (MVP)
* 속성:

  * 위치: 2D 좌표 (x, y)
  * 속도: 벡터 (vx, vy)
* 초기 위치:

  * 경기장 중심 근처 or 랜덤 각도로 발사 (정확 규칙은 튜닝에 따라)
* 초기 속도:

  * v0: “8인 기준 평균 3~5초에 한 번 정도 패들 접촉이 발생하는” 정도의 크기
    (구체 수치는 구현 후 테스트로 조정)

### 5.2 반사 & 이동

* 매 틱(tick)마다:

  * x ← x + vx * dt
  * y ← y + vy * dt
* Arena 벽(정N각형 Side)과 충돌 시:

  * 해당 Side의 법선 벡터를 기준으로 **반사각 계산**
  * 표준 물리 반사: `반사 후 속도 = 속도 - 2*(속도·법선)*법선`

### 5.3 패들과의 충돌

* Ball이 Paddle의 범위와 충돌했다고 판정되면:

  1. **HIT Pang Effect(소형)** 실행
  2. 속도 크기 v에 대해:

     * `v_new = v * 1.05` (5% 증가)
  3. 반사각 계산:

     * 기본: Side에 수직인 법선 기준
     * 추가: 패들의 상대 위치(중앙 기준 좌/우)에 따라 각도 미세 조정 가능(옵션)

### 5.4 OUT 판정 (공을 놓쳤을 때)

* Ball이 특정 Side 영역에서 Paddle과 충돌하지 않고
  **Side를 넘어 Arena 밖으로 나가는** 경우:

  * 해당 Side 플레이어를 OUT 처리.
* 구현 관점:

  * 공이 Side 경계선을 통과했는데,
    그 순간 Paddle 위치가 공과 겹치지 않으면 OUT.

## 6. 공정성 보정 (선택적, 가벼운 보정)

> *목표: “특정 몇 명에게만 공이 몰리는 패턴”을 줄이고,
> 플레이어 모두에게 어느 정도 공이 향하도록 **아주 약하게** 조정.*

### 6.1 히스토리 기반 가중치

* 각 Side마다 `lastHitTime` 기록.
* 최근 T초(예: 8초) 동안 한 번도 Ball이 해당 Side에서 튕기지 않았다면:

  * 다음 반사각 계산 시, 그 Side 방향으로 **극히 미세한 각도 편향** 적용.
* 제한:

  * 각도 조정은 ±2~3도 범위 안에서만.
  * 플레이어가 “조작한다”는 느낌을 받지 않을 정도로만.

> 이 기능은 MVP에서 **ON/OFF 설정 가능**.
> 초기엔 OFF로 시작 후, 필요 시 ON.

## 7. Pang Effect (연출 스펙)

PolyPang이라는 이름에 맞게, **공이 튕기고, 사람이 탈락하는 순간 “Pang”이 난다**는 감각이 중요하다.
아래는 **연출의 수준/길이/강도**까지 포함한 스펙.

### 7.1 HIT Pang (소형) — Ball ↔ Paddle 충돌 시

#### 7.1.1 시각 연출

* 충돌 지점에 작은 스파크/별 조각 이펙트:

  * 반지름: Ball 직경의 1.5~2배 정도
  * 지속 시간: 0.15~0.25초
  * 색상/형태: 밝은 톤, 살짝 튀는 느낌 (구체 색은 아트에서 지정)
* Ball에 짧은 트레일 강화:

  * 충돌 후 ~0.3초 동안 트레일 길이/밝기 증가

#### 7.1.2 사운드

* 짧은 “팅/핑” 계열의 경쾌한 소리:

  * 길이: ~0.15~0.25초
  * 음역: 중~고음, 귀에 피로감을 주지 않는 수준

#### 7.1.3 진동 (옵션)

* 모바일: 아주 짧은 haptic (0.05~0.1초)

### 7.2 OUT Pang (대형) — 플레이어 OUT 시

#### 7.2.1 순서 개요 (타임라인)

1. Ball이 Side를 통과 → OUT 판정
2. **OUT Side 위치에서 Pang 폭발 이펙트** 발생
3. 해당 Side가 깨지며 사라짐
4. Arena 전체가 정(N-1)각형으로 재배치
5. Ball은 0.5초간 살짝 슬로우 상태, 이후 정상 속도 복귀

#### 7.2.2 시각 연출 상세

* OUT Side 상:

  * 폴리곤 형태의 조각들이 바깥쪽으로 흩어지는 이펙트 (0.4~0.6초)
  * Side 라인이 깨어지며 알파→0으로 사라짐
* 중앙/전체:

  * OUT 순간 화면 살짝 흔들림 (카메라 쉐이크, amplitude 매우 얕게)
  * 경기장 전체의 윤곽선이 0.1~0.2초 정도 살짝 진동 후 안정
* Arena 재배치(리메시):

  * 남은 Side들이 새 정(N-1)각형 위치로 **0.3~0.5초 동안 부드럽게 이동**
  * 이동 과정에서 약간의 Ease-in-out 곡선 적용

#### 7.2.3 사운드

* 짧고 명확한 “Pang!” 효과음:

  * “HIT Pang”보다 약간 무겁고, 여운이 0.3초 내외
* 배경음이 있다면:

  * OUT 순간 0.2초 정도 약간 볼륨 Ducking (선택)

#### 7.2.4 UI 텍스트

* 화면 상단 또는 중앙:

  * 예: `P3 OUT!` 또는 `닉네임 OUT!`
  * 노출 시간: 1.0초 이하
  * 2명 이상 동시에 OUT 될 수 있으면 (드물겠지만) 순차 or 리스트로 표기

## 8. 상태 & 플로우 (State Machine)

### 8.1 Room State

* `LOBBY`

  * 인원 입장/퇴장 가능
  * Host가 Start 버튼 사용 가능
* `COUNTDOWN`

  * 3초 카운트다운
  * 인원 변동 없음
* `INGAME`

  * 게임 진행 중
* `RESULT`

  * 게임 결과 표시
  * 이후 로비로 돌아가기 or 나가기 선택

### 8.2 Player State

* `LOBBY_WAIT` : 로비에서 대기/Ready 전 상태
* `LOBBY_READY` : Ready 버튼 클릭 상태
* `INGAME_ALIVE` : 인게임, 아직 탈락 전
* `INGAME_OUT` : OUT 판정 직후, 연출 진행 중
* `SPECTATOR` : OUT 연출 이후, 관전 모드
* `DISCONNECTED` : 연결 끊김 (로비/인게임 공통)

## 9. UX / 화면별 기획 (기획자용)

### 9.1 공통 UI 요소

* 상단:

  * 게임 로고 / PolyPang 텍스트
* 하단:

  * 최소한의 메뉴 (나가기 등)

### 9.2 방 만들기 / 입장 화면

* 요소:

  * [버튼] 방 만들기
  * [입력] 참가코드 입력 (6자리, 영문+숫자)
  * [버튼] 입장
* 동작:

  * 방 만들기 → 서버에 새 Room 생성 요청, 생성 후 로비 화면으로 이동
  * 참가코드 입장 → 유효성 검사 → 성공 시 로비로 이동

### 9.3 로비 화면

* 상단:

  * 참가코드 표시 (예: `CODE: AB3F9K`)
  * [버튼] 코드 복사
* 중앙:

  * 플레이어 리스트 (최대 8명)

    * 닉네임
    * 연결 상태 아이콘
    * Ready 여부 표시
* 하단:

  * [버튼] Ready / Cancel Ready (토글)
  * Host일 경우만 [버튼] Start 활성화

    * 조건: 인원 ≥ 2명
* 플로우:

  * Ready 여부는 UI/서버 상태로 실시간 반영
  * MVP에서는 “모두 Ready 시 자동 시작”은 넣지 않고, Host가 Start를 눌러 시작

### 9.4 카운트다운 화면

* INGAME으로 넘어가기 전 3초 카운트다운:

  * 화면 중앙에 큰 숫자 3 → 2 → 1 → GO
  * Arena 윤곽과 플레이어 위치/닉네임 미리 보이게 해도 됨
* 이 때는 입력을 받아도 Paddle 이동 없음 (입력 무시 or 사전버퍼링 X)

### 9.5 인게임 화면 (플레이어 시점)

#### 9.5.1 상단 HUD

* Alive 플레이어 리스트 (작게)

  * 닉네임 + 색상
  * OUT 된 플레이어는 회색 처리 or 가로줄
* (선택) 현재 공 속도 단계 표시 (예: 속도 레벨 바)

#### 9.5.2 중앙 (Arena)

* 정N각형 경기장 렌더링
* 각 Side에 플레이어 닉네임/색상 표시
* Paddle, Ball, 이펙트 모두 이 영역 수용

#### 9.5.3 하단 입력 UI (모바일)

* 화면 왼쪽 1/3: 왼쪽 이동 영역
* 화면 오른쪽 1/3: 오른쪽 이동 영역
* 중앙 1/3은 터치 무시 영역(실수 탭 방지) or 이모지 버튼 위치로 사용 가능

### 9.6 관전 모드 화면

* 플레이어가 OUT 된 이후:

  * 본인 Paddle/Side 없음
  * Arena는 그대로 보여주고, HUD는 동일
* 추가 UI:

  * 하단에 작은 이모지 버튼들:

    * 👍 😂 😱 🔥 등 4개 내외
  * 이모지 클릭 시:

    * Arena 위에 작은 이모지들이 1~2초 정도 떠다니게
    * 다른 플레이어들도 그 이모지를 볼 수 있음

### 9.7 결과 화면

* 중앙:

  * `WINNER: 닉네임` 크게 표시
* 하단:

  * 모든 플레이어 순위 리스트:

    * 1위~N위, 생존 순서대로
* 버튼:

  * [다시 하기] (Host: 같은 방에서 새 게임 시작, 플레이어: 로비로 이동)
  * [방 나가기] → 처음 화면으로

## 10. 네트워크 & 판정

### 10.1 서버 authoritative

* 공 위치/속도/반사/OUT 판정은 항상 서버가 최종 결정.
* 클라이언트는:

  * Paddle 입력을 서버로 전송
  * 서버에서 받은 상태(Ball, Arena, Player)를 화면에 렌더
  * 필요하다면 예측 렌더링 사용 가능 (세부 구현은 개발 재량)

### 10.2 동기화 주기

* 서버 → 클라 상태 브로드캐스트:

  * 권장: 초당 20~30회
* 클라 → 서버 입력:

  * 입력 이벤트 발생 시 즉시 전송
  * 서버는 일정 간격으로 입력 처리

### 10.3 지연/네트워크 이슈 처리

* 입력 지연:

  * 클라는 자체 예측으로 Paddle 미세 이동 (오차 발생 시 서버 값으로 보정)
* 연결 끊김:

  * 일정 시간 이상 서버와 끊기면 `DISCONNECTED` 처리
  * 인게임 중 끊긴 플레이어는:

    * 일정 타임아웃(예: 5초) 이후 강제 OUT 처리하거나
      그대로 멈춘 Paddle 상태 유지 (정책 선택, 추후 확정)

## 11. 튜닝값 (초기 제안, 개발 후 플레이 테스트로 조정)

* 경기장 반지름 R: 화면 최소 길이의 0.35~0.4
* Paddle 길이 계수 α: 0.3
* Paddle 이동 범위 계수 β: 0.6
* Paddle 최대 이동 속도: (테스트 필요)
* Ball 초기 속도 v0: (테스트 필요, 3~5초에 1회 정도 히트 목표)
* Ball 속도 증가율: +5% (고정)
* HIT Pang 이펙트 길이: ~0.2초
* OUT Pang 이펙트 + 리메시 연출 길이: ~0.8~1.0초
* OUT 슬로우 기간: 0.5초

## 12. 개발 완료 기준 (Acceptance Criteria)

개발/QA가 “MVP 완성”으로 체크할 수 있도록 항목화.

1. **방 & 로비**

   * [ ] 6자리 코드로 방 생성 가능
   * [ ] 코드로 입장 가능 (유효성/인원/상태 검사)
   * [ ] 로비에서 플레이어 목록/Ready 상태 실시간 업데이트
   * [ ] Host만 Start 가능, 2인 이상에서만 시작

2. **경기장 & 플레이**

   * [ ] Alive 수에 따라 항상 정N각형 Arena 생성
   * [ ] 각 플레이어는 1개의 Side를 갖고, 균등 간격 배치
   * [ ] Paddle은 Side 위에서만 자연스럽게 이동
   * [ ] Ball은 Arena 내부에서 물리적으로 반사
   * [ ] Ball-Paddle 충돌 시 속도 5% 증가

3. **OUT & 리메시**

   * [ ] Ball이 특정 Side를 통과할 때 Paddle에 막히지 않으면 해당 플레이어 OUT
   * [ ] OUT 시 OUT Pang 이펙트 + Side 파괴 연출 발생
   * [ ] OUT 후 Arena는 정(N-1)각형으로 재배치
   * [ ] OUT 플레이어는 Spectator로 전환
   * [ ] Alive가 1명 남으면 게임 종료 & 결과 화면 표시

4. **Pang 연출**

   * [ ] Ball-Paddle 충돌 시 HIT Pang 이펙트/사운드 재생
   * [ ] OUT 시 OUT Pang 이펙트/사운드 재생
   * [ ] 연출 길이 때문에 게임 템포가 과도하게 느려지지 않음

5. **관전 & 이모지**

   * [ ] OUT 이후 관전 화면으로 전환, Arena를 그대로 볼 수 있음
   * [ ] 관전자는 이모지 버튼을 눌러 리액션을 보낼 수 있음
   * [ ] 이모지는 모든 클라이언트 화면 상에 잠시 표시됨

6. **플랫폼 & 안정성**

   * [ ] 모바일/PC 모두에서 입력 & 플레이 가능
   * [ ] 서버 authoritative 방식으로 판정 일관성 유지
   * [ ] 기본적인 끊김/재연결 케이스에서 치명적 버그 없음

---

여기까지만 구현되면, **PolyPang MVP 1판이 실제로 돌아가는 상태**라고 보면 돼.
이제 이 문서를 **“룰/플로우 기준”으로 고정하고**, 다음 단계에서:

* 화면기획자: 화면별 구조/와이어/컴포넌트 구체화
* 개발자: 엔진/프레임워크 선택 + 엔티티/메시 구조 설계

## 13. 엣지 케이스 및 정책 (Edge Cases & Policies)

개발/QA 시 발생할 수 있는 엣지 케이스에 대한 **명확한 정책**을 정의.

### 13.1 연결 끊김 및 재접속 정책

| 상황 | 타임아웃 | 처리 방식 |
|------|----------|-----------|
| **LOBBY 중 끊김** | 30초 | 리스트에 "연결 끊김" 표시 <br> 30초 내 복귀 시 복원 <br> 30초 초과 시 자동 퇴장 |
| **INGAME 중 끊김** | 5초 | 패들 "마지막 위치 고정" <br> 5초 내 복귀 시 ALIVE 유지 <br> 5초 초과 시 **즉시 OUT 처리** (Side 제거/리메시) |
| **RESULT 중 끊김** | 즉시 | 자동 퇴장 (결과는 이미 확정됨) |

### 13.2 OUT 판정 기준 (수학적 조건)

| 조건 | 판정 기준 |
|------|-----------|
| **Ball 위치 기준** | Ball 중심이 Side 선분을 넘어간 순간 |
| **패들 히트 우선순위** | 1. Paddle 범위 내 → `PADDLE_HIT` <br> 2. Paddle 범위 밖 선분 → `WALL_REFLECT` <br> 3. 선분 완전 통과 → `SIDE_OUT` |
| **코너 근처 충돌** | 정점에서 ±5° 이내는 두 Side 중 가까운 쪽으로 판정 |

### 13.3 공 초기 발사 규칙

| 항목 | 규칙 |
|------|------|
| **초기 위치** | Arena 중심 (0, 0) |
| **초기 방향** | 서버 Seed 기반 난수로 각도 θ₀ 선택 <br> **제약**: 모든 플레이어 Side 중심각에서 ±15° 이상 떨어진 값만 허용 <br> → 시작 0.5초 내 "한 명에게 직격" 방지 |
| **초기 속도** | v₀ (튜닝값, 섹션 11 참조) |

### 13.4 Host 변경 정책

| 상황 | 처리 방식 |
|------|-----------|
| **LOBBY 중 Host 퇴장** | 1. 가장 먼저 입장한 Alive 플레이어가 자동 Host 승계 <br> 2. 모든 플레이어에게 `host_changed` 이벤트 전송 <br> 3. UI에 "Host 변경" 토스트 표시 (1초) |
| **INGAME 중 Host 퇴장** | Host OUT 처리 후, 자동 승계 (게임 진행에 영향 없음) |
| **마지막 플레이어 퇴장** | 방 자동 삭제 |

### 13.5 동시 OUT 처리

| 상황 | 처리 방식 |
|------|-----------|
| **2명 이상 동시 OUT** | 1. 서버는 tick 순서대로 처리 (순차) <br> 2. 각 OUT마다 `out_pang` + 리메시 진행 <br> 3. 클라이언트는 0.2초 간격으로 연출 재생 (시각 난잡 방지) |
| **Alive 0명 (무승부)** | 마지막 동시 OUT 시: <br> - 가장 오래 생존한 플레이어를 우승자로 판정 <br> - 동률이면 먼저 입장한 플레이어 우승 |

### 13.6 공정성 보정 활성화 기준

| 항목 | 기준 |
|------|------|
| **활성화 조건** | 내부 QA에서 "최근 10초 동안 특정 2인에 Ball 접촉 70% 이상" 패턴이 <br> 3회 이상 관측되면 → 보정 활성화 검토 |
| **배포 초기** | OFF (기본값) |
| **A/B 테스트** | 유지율 낮으면 ON 검증 |
